# cat-room 长期操作流程（总控 / 施工 / 审核）

## 0. 这份文档是干什么的

这是 `cat-room` 项目的长期操作流程。  
以后每次开新聊天，不要默认助手记得当前这次具体任务；但也不用从零重讲整个项目。

这个项目里默认分工如下：

- **ChatGPT（助手）**：总控。负责理解当前阶段、讨论这次任务、给出操作步骤、生成“给 Codex 的施工指令”和“给审核员的验收指令”。
- **Codex web**：施工。负责读取 GitHub 仓库、执行小任务、产出 diff、创建 PR。
- **GitHub PR**：验收中枢。所有改动尽量都经过 PR。
- **Codex review**：第一层审稿。
- **Gemini Code Assist on GitHub**：第二层审稿。

---

## 1. 助手默认行为

当用户在这个项目里请求操作指导时，默认按 **0 基础用户** 处理。

要求如下：

1. 不要省略入口网址。
2. 涉及网页操作时，第一步先说明去哪个网址。
3. 每一步都写清楚：当前页面、要点哪个按钮、按钮的中文名和英文原名。
4. 不要默认用户理解 GitHub、PR、分支、Conversation、Files changed 等概念。
5. 能一步一步教，就不要只讲概念。
6. 除非用户明确要求简略版，否则默认写成傻瓜式操作。

---

## 2. 每次新任务时，用户怎么开头

以后每次开新聊天，默认先发“开场包”：

**仓库链接：**  
【GitHub 仓库链接】

**关键文档 / 索引：**  
【docs 索引、施工规划、决策文档链接】

**当前阶段：**  
【例如：重构规划 / 第一阶段施工 / 文档整理 / PR 修正】

**这次任务：**  
【这次具体要做什么】

**不要动：**  
【这次明确不能碰的部分】

**补充说明：**  
【例如：这是 GitHub Pages 项目；优先小步改动；尽量不改现有玩法】

---

## 3. 助手收到开场包后，要先做什么

助手先判断任务属于哪一类：

### A. 需要先讨论 / 先规划的任务

如果这次是重构、牵涉面大、边界不清楚、可能动很多文件，就不要直接施工。  
先讨论，必要时先给一份 **只做规划、不改代码** 的 Codex 指令。

### B. 可以直接施工的任务

如果目标很清楚、范围很小、边界明确，就直接给施工指令。

---

## 4. 助手每次最终要给出的两份内容

每次讨论结束后，默认给用户两份内容：

### ① 给 Codex 的指令

用于去 Codex web 直接发任务。

### ② 给审核员的指令

用于 PR 审稿。  
如果仓库里已经装好了 Gemini Code Assist on GitHub，这份内容主要是“在 PR 评论区该发什么命令、看哪里、怎么判断结果”。

如果某次任务太复杂，助手可以先不给施工指令，而是先给 **Codex 规划指令**。

---

## 5. Codex 的使用规则

### 5.1 什么时候先 `/plan`

满足任一情况，就优先先规划：

- 这是重构，不是小修
- 用户不确定该先动哪一块
- 这次可能会改多个文件
- 这次目标是“建立流程 / 拆分职责 / 重新整理结构”

### 5.2 什么时候可以直接施工

满足大多数情况时，可以直接施工：

- 这次只有一个明确目标
- 能说清楚“别动什么”
- 涉及文件不多
- 很容易验收

### 5.3 给 Codex 的长期固定要求

以后给 Codex 的施工指令，默认都带上这些约束：

1. 只做这一次任务直接相关的内容
2. 不要顺手重构整个项目
3. 不要新增复杂依赖
4. 尽量保持现有玩法、表现、文档口径不变，除非任务明确要求修改
5. 改完后必须说明：
   - 改了哪些文件
   - 为什么这么改
   - 用户该怎么验证
6. 如果能运行验证，优先验证

---

## 6. 施工标准流程

### 第一步：先在项目里和助手讨论任务

用户先发“开场包”。  
助手负责判断：先讨论、先 `/plan`，还是直接施工。

### 第二步：拿着助手给的 Codex 指令，去 Codex web

先确认仓库和分支选对，再发任务。

### 第三步：Codex 跑完以后，先不要急着合并

先看三件事：

1. 改了哪些文件
2. 它自己说做了什么
3. diff 有没有超范围

如果明显乱改，就不要继续推进，先回来收紧任务。  
如果看起来正常，再创建 PR。

### 第四步：创建 PR

Codex 跑完以后，在 diff view 审一下改动，再创建 PR。  
能开 Draft PR 时，优先开 Draft PR。

---

## 7. PR 审稿标准流程

### 第一层：Codex review

在 GitHub PR 页面里，进入 **Conversation** 页签。  
在评论框输入：

`@codex review`

然后点 **Comment**。  
不要点 **Close with comment**。

> 说明：Codex review 更偏向抓严重问题，文档排版或小结构问题不一定会提。

### 第二层：Gemini Code Assist review

如果仓库已经安装了 **Gemini Code Assist on GitHub**，以后优先直接在同一个 GitHub PR 里调用 Gemini。

在 PR 评论区常用命令：

- `/gemini review`
- `/gemini summary`

建议顺序：

1. 先看 Gemini 有没有自动给 review
2. 如果没动静，就手动发 `/gemini review`
3. 想先看概述，就发 `/gemini summary`
4. 看完意见后，再决定要不要修

---

## 8. 什么时候可以合并

满足下面这些条件时，才建议 merge：

1. 用户自己看过 diff，没有明显超范围
2. `@codex review` 没提严重问题
3. Gemini review 没提严重问题
4. 这次任务的目标已经完成
5. 没有多余的顺手改动

如果其中一层指出问题，就不要硬合并。  
先把问题带回助手窗口，生成“修正单”，再开下一轮小改动。

---

## 9. 每次合并后，用户该怎么回来汇报

合并后，回到项目里开一个新聊天，默认发下面这段：

**本次已合并：**  
【PR 链接】

**本次完成内容：**  
【一句话概括】

**当前状态：**  
【例如：第一阶段已完成 / 文档索引已同步 / 下一步准备拆分输入逻辑】

**请根据当前阶段告诉我下一步最合适做什么。**  
如果适合继续施工，请直接给我：

1. 给 Codex 的执行指令
2. 给 PR 审稿用的指令

---

## 10. 助手输出格式要求

以后当用户说“给我这次的施工单”时，默认按下面格式输出：

### A. 这次任务判断

一句话说明：  
这是“先讨论 / 先 `/plan` / 可直接施工”中的哪一种。

### B. 给 Codex 的指令

直接给可复制版本。

### C. 给 PR 审核的操作

直接告诉用户：

1. 先发 `@codex review`
2. 再看 Gemini 自动 review 或手动发 `/gemini review`
3. 去哪一页看结果（Conversation / Files changed）

### D. 如果失败，下一步怎么办

如果某一步没反应，也要直接给排查顺序，不要只讲概念。

---

## 11. 用户最短只需要记住的一句话

**先在项目里找助手定任务 → 去 Codex web 施工 → 创建 PR → 在 PR 里发 `@codex review` → 再看 Gemini 自动 review 或手动发 `/gemini review` → 没问题就 merge → 回项目里领下一单。**
